# Biblioteca â€” Code Review (autonomous pass)

Scope: entire `src/` tree, file by file. Prior `tot.txt` declared "all per-file
findings resolved", so this is a fresh, independent audit. Findings below were
re-verified against current code (line ranges are current). Files not listed were
judged correct enough to skip.

**[src/presentacio/util/EstatVistaBiblioteca.java]** â€” CRITICAL
- `public final Deque<Llibre> undoBuffer = new ArrayDeque<>();` (line 27) is a
  non-thread-safe collection that is mutated from background `SwingWorker`
  threads (see `ControladorMenuContextual` / `ControladorAccionsLlibre`) and
  read/popped on the EDT (`undoDelete`, line 181). `ArrayDeque` gives no happens-before and
  is not safe for concurrent `push`/`removeLast`/`pop`; two concurrent batch
  deletes (e.g. gallery delete + table delete) corrupt internal head/tail
  indices â†’ lost undo entries, `NullPointerException`, or an inconsistent size.
  Root cause: shared mutable state crossing the EDT/worker boundary without
  synchronization. Fix: use `java.util.concurrent.ConcurrentLinkedDeque` (and an
  `AtomicInteger`/`size()` cap), or confine ALL mutations to the EDT by pushing
  inside `done()` instead of `doInBackground()`.

**[src/presentacio/controladors/ControladorMenuContextual.java]** â€” CRITICAL
- `eliminarLlibresGaleria` (178-222) calls `state.undoBuffer.push(l)` and
  `state.undoBuffer.removeLast()` (lines 194-195) inside `doInBackground()` â€”
  i.e. off the EDT â€” on the shared `ArrayDeque`. This is one of the two writer
  sites of the undo-buffer data race. Fix: collect the deleted `Llibre`s in the
  worker and push them onto the deque in `done()` (EDT), or switch the field to a
  concurrent deque.

**[src/presentacio/controladors/ControladorAccionsLlibre.java]** â€” CRITICAL, HIGH
- CRITICAL: `state.undoBuffer.push(l)` / `removeLast()` (lines 130, 137) run
  inside `doInBackground()` while `undoDelete()` pops on the EDT â€” concurrent
  access to the non-thread-safe `ArrayDeque` (same race as above). Fix as above.
- HIGH: `mostrarEstadistiques` (~192) performs CPU-heavy aggregation plus several
  synchronous data fetches directly on the EDT; for a large library this freezes
  the UI. Fix: compute in a `SwingWorker.doInBackground()` and only build the
  dialog in `done()`.
- (Covered by DelegatLlibre fix) line 364 calls `state.cd.esLargeLibrary()` in a
  worker's `done()` (EDT), which runs a `SELECT COUNT(*)`.

**[src/domini/facade/DelegatLlibre.java]** â€” HIGH
- `esLargeLibrary()` (92-93) â†’ `comptarLlibresDB()` (237) â†’ `persistence()
  .comptarLlibres()` issues `SELECT COUNT(*) FROM llibre` on every call. The full
  library is already resident in memory: `ControladorDomini` loads it at
  construction (`new ArrayList<>(cp.obtenirAllLlibres())`, ControladorDomini:82)
  and `getSize()` (88-90) already returns `state.bib().size()`. These methods are
  invoked on the EDT from at least 8 sites â€” `ControladorAccionsLlibre:364`,
  `ControladorPrestatgeria:69/85/90`, `ControladorPaginaTaula:65/70`,
  `ControladorFiltre:148`, `ControladorMostrarBiblioteca:180` â€” so every list
  selection, filter, page change and refresh blocks the EDT on a JDBC round-trip
  for a number it already has in RAM. Fix: implement
  `comptarLlibresDB()` as `state.withLockReturning(() -> state.bib().size())` and
  derive `esLargeLibrary()` from it; this removes JDBC from all EDT call sites at
  once.

**[src/domini/ControladorDomini.java]** â€” MEDIUM
- The constructor (80-85) eagerly loads the ENTIRE `llibre` table into
  `state.bib()` unconditionally. This contradicts the whole "large library"
  subsystem (`esLargeLibrary` â†’ DB pagination via `obtenirLlibresPage` /
  `comptarLlibresDB`): if every book is already in memory, the DB-pagination path
  is dead weight; if a library is genuinely large, the startup `obtenirAllLlibres`
  is the real scalability problem, not the `COUNT`. Pick one strategy â€” either
  keep the in-memory model and serve counts/pages from `bib`, or make loading
  lazy and never materialize the whole list. Today both coexist inconsistently.

**[src/persistencia/dao/LlibreDaoCore.java]** â€” HIGH, MEDIUM
- HIGH (wrong query result): `obtenirRecentlyAdded` (213-232) builds its SQL from
  `LLIBRE_COLUMNS` (line 112), the unprefixed projection. `buildSelect` strips the
  alias with `frag.replace("l.", "")` (line 134). Applied to the correlated author
  subquery (line 48,
  `... WHERE la.isbn = l.ISBN) AS autor`) this yields `... WHERE la.isbn = ISBN`.
  In the subquery scope the only `isbn`-named column belongs to the inner
  `llibre_autor la`, so unqualified `ISBN` binds to `la.isbn` (SQL is
  case-insensitive), making the predicate `la.isbn = la.isbn` â€” a tautology. The
  correlation to the outer `llibre` row is lost, so `GROUP_CONCAT` returns EVERY
  author in the database for EVERY recently-added row. The in-code comment
  (216-220) reasons only about column-name lookup and misses the broken
  correlation; the doc comment at 110-111 even claims `LLIBRE_COLUMNS` has no
  consumer ("cap avui") and marks it `@SuppressWarnings("unused")`, both untrue.
  Fix: use the aliased form like `obtenirAll` â€”
  `"SELECT " + LLIBRE_COLUMNS_L + " FROM llibre l ORDER BY l.data_afegit DESC LIMIT ?"`
  â€” then delete the now-truly-unused `LLIBRE_COLUMNS` and its stale comment/suppression.
- MEDIUM: every public method is `synchronized` (192, 213, 234, 249, â€¦) even
  though all access funnels through `ControladorPersistencia`, which serializes on
  its own monitor, and sibling DAOs (`LlistaDao`, `AutorDao`) explicitly document
  that DAO-level locking is unnecessary. The redundant `synchronized` is
  inconsistent and misleads maintainers into thinking the DAO is independently
  thread-safe. Same applies to `TagDao`, `LlibreBlobDao`, `LlibreLecturaDao`.
  Fix: drop `synchronized` from the DAOs and centralize the contract in one
  Javadoc on `ControladorPersistencia`.

**[src/persistencia/dao/AutorDao.java]** â€” MEDIUM

**[src/herramienta/io/ServeiCopiaSeguretat.java]** â€” HIGH, MEDIUM
- HIGH (backup integrity): `copiaSegToSQL` (~179) snapshots the main book rows but
  then reads the relational tables (authors, lists, tags, loans, lectures) live,
  outside the state lock, one DAO call at a time. While the auto-backup scheduler
  runs (`scheduleAutoBackup`, started from `ControladorDomini:94`) the user can be
  editing concurrently, so the dump can contain `llibre_autor`/`llibre_llista`
  rows referencing an ISBN that is not in the book snapshot â†’ dangling FK in the
  `.sql`. On restore the FK-bearing `INSERT` fails and the whole transaction rolls
  back (restore aborts) â€” a silent "my backup won't restore" failure. Fix: take
  the entire snapshot (books + all relations) under a single
  `state.withLockReturning(...)`, or run the dump inside one repeatable-read JDBC
  transaction so all reads see one consistent point in time.
- MEDIUM (fidelity): `sqlEsc` (~376) replaces newlines with spaces, so multi-line
  text fields (description/notes) silently lose their line breaks on a
  backupâ†’restore round-trip. Fix: emit `\n` as `'...' || CHAR(10) || '...'` or
  keep the literal newline and make the restore splitter (which already tracks
  quote state) preserve in-quote newlines.

**[src/domini/facade/DelegatCopiaSeguretat.java]** â€” HIGH
- `copiaSegToSQL` (~47) takes `bib`/`llistes`/`tags` snapshots under
  `state.lock()` but then hands off to `ServeiCopiaSeguretat`, which reads the
  relational rows live without the lock (see finding above). The partial locking
  gives a false sense of atomicity. Fix: snapshot the relational rows under the
  same lock and pass them in, instead of letting the service re-read them.

**[src/herramienta/io/ServeiCoberta.java]** â€” MEDIUM
- Lock granularity: `obtenirCachedBytes` (63-79) holds `L1_LOCK` across
  `Files.isRegularFile` + `Files.readAllBytes` (lines 69-70), so a single L2 disk
  read serializes every other thread touching the cover cache (`putL1`,
  `cacheBytes`, other reads). It is reachable on the EDT via
  `ControladorExportacio.fetchMissingCovers` â†’ `submitCoverFetch` â†’
  `enqueueFetch` (which calls `obtenirCachedBytes` synchronously, line 117), so a
  cache hit can perform disk I/O under a global lock on the EDT while six
  `WRITE_POOL` threads contend the same lock. Fix: double-check â€” look up L1 under
  the lock, release it, read the disk outside the lock, then `putL1` under the
  lock.
- Dead public API: `fetchAsync` (92), `obtenirCachedImage` (81) and the
  `@Deprecated POOL` alias (48) have no callers. Remove them.
- `cacheBytes` (164-177) swallows disk-write failures with an empty
  `catch (Exception ignored)` (176) while the rest of the class logs at FINE;
  inconsistent and hides a full/again-read-only cache dir. Log it.

**[src/presentacio/controladors/ControladorExportacio.java]** â€” MEDIUM

**[src/herramienta/text/ValidadorLlibre.java]** â€” MEDIUM

**[src/presentacio/galeria/ServeiImatgesCoberta.java]** â€” MEDIUM
- The LRU `imageCache` is an access-ordered `LinkedHashMap` (`accessOrder=true`,
  57-58), but `obtenirCached` (85-92) and `submit` (120-129) call
  `imageCache.get(isbn)` holding only the READ lock. On an access-ordered map
  `get()` relinks the node (`afterNodeAccess`) â€” a structural mutation â€” so the
  concurrent readers the shared read lock permits would corrupt the list, making
  the class's documented "concurrent EDT reads" guarantee (Javadoc 35-41) false.
  Latent only because every caller currently runs on the EDT. Fix: take the write
  lock in `obtenirCached`/`submit`, or set `accessOrder=false` and track recency
  separately, and correct the Javadoc.

## Calibration

1. Weakest finding: the `ServeiCoberta` lock-across-I/O (MEDIUM). The reasoning
   that it stalls the EDT depends on the `fetchMissingCovers` path hitting an L2
   *cache hit* (the only branch that calls `Files.readAllBytes` under the lock);
   in that flow the covers are by definition missing, so most calls only do a
   `Files.isRegularFile` stat under the lock. Stronger evidence would be an EDT
   sample/stack capture showing time parked on `L1_LOCK` during a real
   `readAllBytes`, or a microbenchmark of cache-access latency under WRITE_POOL
   contention. The lock-granularity defect itself is real regardless.
2. New class to look for next run: this run surfaced a recurring pattern â€” NPEs and
   `NumberFormatException`s from unboxing/stringifying nullable boxed getters
   (`obtenirISBN():Long`, `obtenirAny():Integer`, `obtenirLlegit():Boolean`) at
   `imprimirFitxa`, `EditorCasellaLlegit`, and `LligadorFormulariLlibre`. Next run
   I will do a dedicated codebase-wide sweep for every unguarded use of those
   nullable getters. Still un-audited: (a) dynamic `ORDER BY`/sort-column
   whitelisting in `LlibreSearchDao` for SQL-injection via user-chosen sort keys,
   and (b) i18n key coverage (hardcoded user-facing strings / missing `I18n.t`).
3. Rejected this run (verified, then dropped to avoid false positives): the
   "restore corrupts on `;`/`--` inside data" idea â€” `processarLine` correctly
   tracks quote state, so only multi-line literals lose newlines, which `sqlEsc`
   already prevents; and the "cover write sets has_blob=true with a null blob"
   idea â€” `has_blob` is a computed subquery `(imatge_blob IS NOT NULL)`, so a null
   blob simply reports `false`.
