# Biblioteca — Code Review (2026-06-13)

**[src/domini/NamedEntity.java]** — [severity: LOW]
- The interface is a 2-method read-only view. The two implementers
  (`Llista`, `Tag`) both expose `getId()` and `getNom()` publicly,
  so the interface is purely for the type system. Used only by
  `EtiquetasDelLlibreDialog.updateComboAdd` path through
  `EntityComboUtils` (line 11) — but `EntityComboUtils` takes a
  raw `ToIntFunction<E>`, not a `NamedEntity`. The interface
  itself is dead.

**[src/domini/SortSpec.java]** — [severity: LOW]
- `SQL_COLS` (lines 22-28) and `COMPARATORS` (lines 39-45) are two
  parallel maps. Adding a new column requires editing both
  (plus the Javadoc) and a new comparator class would re-use
  `ISBN_COMPARATOR` patterns. Consider a single
  `record Column(String key, String sqlFragment, Comparator<Llibre> cmp)`
  with a static `Map<String, Column> COLUMNS = Map.of(...)`.
- `comparator(String)` (lines 69-71) returns the comparator or
  null. The call site in `BookDelegate.applySort` (line 232) maps
  null to `ISBN_COMPARATOR`. The fallback should live in
  `SortSpec` to keep the policy in one place.

**[src/persistencia/AutorDao.java]** — [severity: LOW]
- `createAutor` (lines 34-46) returns the generated id or
  throws if the driver returns no keys. Throwing a
  `BibliotecaException` (not `SQLException`) loses the
  underlying cause context. Wrap as `new BibliotecaException(
  "Error creant autor: " + e.getMessage(), e)` to preserve
  the stack.

**[src/persistencia/LlibreBlobDao.java]** — [severity: LOW]

**[src/herramienta/BackupService.java]** — [severity: MEDIUM]
- `autoBackup` (lines 59-84) is scheduled on a daemon
  single-thread `ScheduledExecutorService` (line 41). If
  the backup takes longer than the 24h interval
  (line 33), two backups run concurrently. The fix is
  either `scheduleWithFixedDelay` (already used) which
  waits for completion, or check `scheduler.awaitTermination`
  before submitting. The current
  `scheduleWithFixedDelay` (line 46) is the correct
  primitive — so concurrent invocations are impossible.
  But the local `File[] existing = dir.listFiles(...)` on
  line 70 races with the auto-prune on line 79; if a backup
  is in flight when the prune runs, the prune can delete
  the in-progress file. Use a sequence number suffix or
  serialize access via the executor.
- `writeLlibreINSERT` (lines 121-154) builds the INSERT
  via `StringBuilder.append` with 27 hand-typed columns.
  The class Javadoc for `BackupTablePrinter` (line 5)
  flags this as future-refactor target. Today the printf
  duplicates the column list from
  `LlibreFieldBindings.forInsert`. If a column is added
  to one, the other is forgotten. Centralize in
  `LlibreFieldBindings` (it already has `forInsert`) and
  generate the SQL via `String.format("INSERT INTO llibre
  (%s) VALUES (%s);", colList, placeholders)`.
- `backupToSQL` (lines 86-119) opens a `PrintWriter` over
  a `FileWriter` in a try-with-resources — good. The
  inner `try` blocks inside the for-loops (none here, but
  the `autoBackup` method has one on line 81) catch
  `Exception` and `LOG.log(Level.WARNING, "...")`.
  Swallowing all exceptions hides bugs. Distinguish
  `IOException` (log and continue), `SQLException`
  (propagate or rethrow), and runtime exceptions
  (propagate).
- `shutdownScheduler` (line 50) sets `schedulerStarted`
  to `false` (line 56). If `scheduleAutoBackup` is
  re-called after a shutdown, the new scheduler is
  created and `scheduleWithFixedDelay` is re-scheduled.
  The two consecutive calls produce a non-running
  scheduler because `shutdownNow` is interrupt-driven
  and may take time to settle.


**[src/herramienta/I18n.java]** — [severity: LOW]
- The class is 628 lines of `T(key, ca, es, en)` calls
  (lines 14-587). The auto-gen comment (line 1)
  documents that `strings/strings.csv` is the source.
  But the generated file is checked into source, so
  drift is possible (a manual edit to I18n.java would
  be overwritten on next regen). A pre-commit hook or
  CI check that the file matches the CSV would catch
  drift.
- `t(key, args...)` (line 593) checks
  `!raw.contains("{0}")` (line 606) and logs a warning
  to stderr (line 607). Two problems: (a) the warning
  goes to stderr unconditionally — in production, this
  clutters the user's terminal; (b) the check is at
  position 0 only; a key with `{1}` but no `{0}` is
  not flagged. Either check all positions or remove
  the check.
- `langIndex` (line 622) maps "es"→1, "en"→2, "ca"→0.
  Anything else returns 0 (ca). Adding a new language
  (e.g. "fr") silently falls back to ca. Throw or
  log a warning for unknown codes.

**[src/herramienta/Isbn13Normalizer.java]** — [severity: LOW]
- `toIsbn13` (line 8) returns the original `digits`
  string if the length is not 10 or 13 (line 17). The
  caller `LlibreValidator.checkLlibre` then calls
  `Long.parseLong(digits)` which throws on non-numeric
  input. The fallback is silent — the user sees a
  validation error with the wrong field. Either throw
  with a clear message or return null and let the
  caller decide.

**[src/herramienta/Isbn13Normalizer.java]** — [severity: LOW]
- `toIsbn13` line 12: `trimmed.replaceAll("[^0-9X]", "")` —
  strips anything that isn't a digit or X. This
  converts "ISBN: 978-..." into "978..." but also
  drops non-ASCII characters silently. If a user
  pastes "ISBN 9780" with a non-ASCII space, the
  result is "ISBN9780" — note the "ISBN" prefix is
  stripped silently. A user pasting an ISBN from a
  PDF with a space-separated prefix would see the
  wrong number.

   was flagged as LOW but the evidence is thin: I noticed
   `chosen.trim().replaceAll("[^a-zA-Z0-9_-]", "_")` strips non-ASCII
   characters from a user-typed profile name silently, but I did
   not test whether the resulting profile name actually
   round-trips through `H2` or breaks a re-connection flow. To

   that `Config.getDbProfile()` returns "caf_", and that
   re-opening the dialog shows "caf_" (not "café"). The
   concrete failure mode (silent rewriting) is plausible but
   unconfirmed; the fix is trivial (add a warning dialog).
2. **New class of issue to look for next run.**
   I missed auditing the **SQLite-on-Calibre** integration
   end-to-end. The shell-out in `BookImporter.importCalibre`
   passes a multi-MB SQL through `sqlite3` stdout and
   parses with `line.split("\\|", -1)` — but I did not
   test with a 1000-book Calibre library to see whether
   the process timeout (30s, line 99) is sufficient, nor
   did I check the charset handling for non-UTF8 Calibre
   libraries. The pipe-delimited SQL also assumes no book
   title contains a literal `|` character, which is rare
   but possible. A next-run test fixture with a Calibre
   library containing `|`-titled books would expose the
   parser's blind spot.
