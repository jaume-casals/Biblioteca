 5.  ANTI-PATTERNS  (status: all OPEN unless noted)
================================================================================

[HIGH]  Singleton god-object — ControladorDomini — STILL OPEN.
        60+ public methods, in-memory ArrayList<Llibre|Llista|Tag>, only
        path from UI to data.  Documented in Javadoc; sub-facade for
        navigation would help but not started.

[HIGH]  Singleton god-object — ControladorPersistencia — STILL OPEN.
        Previous "double-locking" critique was wrong: methods are now
        uniformly synchronized on the controller, and DAOs do not
        double-lock.  Facade is fine, but no sub-facade for navigation.

[HIGH]  Mutable statics — UITheme.BG_MAIN, ACCENT, etc. — STILL OPEN.
        Theme switching reassigns these fields at runtime so they can't
        be `final`.  Accepted debt; a Map<Theme, Palette> refactor would
        touch 273+ call sites.  Note: UITheme.Palette.java already
        documents the migration plan; switch in setTheme remains.

[HIGH]  Anemic domain model — Llibre — PARTIALLY OPEN.
        Llibre now has a Builder, bindUpdateableFields(), addAutorNom,
        setAutors(), and `isHeavyFieldsLoaded` lifecycle.  Still mostly
        getter/setter symmetry on ~25 fields; the heavy behaviour
        (validation, persistence orchestration) lives in LlibreValidator
        and ControladorDomini.

[MED]   Inverted dependency — Llista.setColor / ControladorDomini.setLlistaColor
        uses FQN `herramienta.I18n.t` rather than an import.  Coupling is
        real but minimized.  Full fix would move the i18n to the UI caller.

[MED]   Inverted dependency — BackupService (herramienta) uses
        ControladorPersistencia.  Util reaching into persistence.
        Accepted debt.

[MED]   Layer inversion — presentacio/detalles/control/ vs presentacio root
        vs presentacio/listener/ — inconsistent sub-packages.
        Accepted debt.

[LOW]   Manual re-implementation of Properties in Config.props — STILL OPEN.
        ConcurrentHashMap-based props are read often (every Config getter
        call) and the boot path stays simple; java.util.Properties would
        need explicit synchronization for the same use.  Accepted debt.
