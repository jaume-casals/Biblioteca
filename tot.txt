# Biblioteca — Code Review (2026-06-13)

Autonomous review pass. Every file in `src/` was read. Findings below are
ranked CRITICAL → LOW. Each file's section is capped at three findings; an
"overflow bucket" is never used.

---

**[src/domini/facade/BookDelegate.java]** — [severity: HIGH]
- `addLlibre` / `deleteLlibre` / `deleteLlibreByIsbn` / `updateLlibre` /
  `setLlibreBlob` (lines 96-189) follow a "binarySearch → DB → in-memory
  mutate" pattern that is not atomic with respect to the lock. Two threads
  inserting the same ISBN can both pass the binarySearch before either
  commits; on the second insert the DB throws, but the surviving add can
  then insert at a `pos` that was computed before the other thread's
  in-memory mutation. The class Javadoc (lines 32-36) acknowledges the
  window. Fix: re-acquire the lock between the DB call and the list
  mutation, and re-validate presence/absence inside that critical section;
  or wrap each call in a single `synchronized` and re-do the binary search
  on the second pass. The known-window Javadoc is fine for a personal
  single-JVM app, but the consequence in `setLlibreBlob` is worse: a
  null/negative search result is silently treated as "book not in list,
  skip" so two writers each only persist their own blob.
- `getLlibre(long)` (lines 145-157) loads heavy fields OUTSIDE the
  state lock via `loadHeavyFields(l)`. While the read is by ISBN, a
  concurrent `updateLlibre` can mutate `l` between the lock release and
  the heavy-load, so the caller's `l` may be partially overwritten
  mid-loadHeavyFields (descripcio/notes set after the first update but
  before the second). The fix is either (a) keep the in-memory reference
  pinned in `bib()` (look up by ISBN after load to swap the slot), or
  (b) make `loadHeavyFields` take a defensive copy and only assign back
  if the book is still at the same row.
- The pre-`SQL_FILTER_THRESHOLD` short-circuit at line 195
  (`if (state.bib().size() >= SQL_FILTER_THRESHOLD) return
  searchLlibresSQL(f);`) is read twice: once via `aplicarFiltres(f)` and
  again by every DB-backed caller. The 2000 magic number is hard-coded
  in two places (`BookDelegate.SQL_FILTER_THRESHOLD` and the comment
  in `TablePageController`). Extract to a single named constant on
  `ControladorDomini` so the threshold can be tuned or tested without
  a global search.

**[src/domini/facade/BackupDelegate.java]** — [severity: HIGH]
- Lines 46-50: heavy-field preloading is performed on the caller's
  thread (the Swing EDT, since `BackupController.backupBD` invokes
  `cd.backupToSQL(f)` from `SwingWorker.doInBackground` — but the
  constructor wiring and any other call sites run on the EDT directly).
  For a 5,000-book library each missing heavy fields, this is 5,000
  sequential `SELECT descripcio, notes FROM llibre WHERE ISBN = ?`
  round-trips on the EDT. Fix: keep the pre-load step on the calling
  thread but batch the lookups via a single `SELECT ... WHERE ISBN IN
  (...)`, or only call `loadHeavyFields` lazily for books that
  serialize descripcio/notes.
- `restoreFromSQL` (lines 63-100) builds `newBib` INSIDE the lock
  (lines 92-98) by calling `state.persistence().getAllLlibres()` etc.
  on the thread that holds `state.lock()`. JDBC `getAll*` queries
  executed under the application state lock serialize all other
  delegate operations; if the DB is slow, every other read/write
  (including table filter, search, shelf add) blocks. Acquire
  snapshots via a dedicated persistence method that does NOT touch
  the state lock, then publish atomically.
- `backupToSQL` lines 56-58 wrap the snapshot creation in
  `synchronized (state.lock())`. A pre-existing exception path leaves
  `state.llistes` partially populated only if the same exception path
  is exercised. Today the try-block has no inner try, so the lists
  are fully populated before being handed off. Fine in practice, but
  the Javadoc does not mention that the snapshot is per-collection
  rather than transactional — a slow DB on the first
  `getAllLlibres()` is fine, but a slow DB on the last one
  (`getAllLlistes`) keeps the lock for the full duration.

**[src/domini/facade/StateContext.java]** — [severity: MEDIUM]
- `bib()`, `llistes()`, `tags()` return the live `ArrayList`. The
  Javadoc (lines 53-61) is correct that callers MUST hold the lock,
  but the contract is unenforced. Consider making the backing
  collections immutable and re-publishing on every mutation, or
  returning a defensive `Collections.unmodifiableList` view that
  fails fast on mutation attempts. The current pattern means a
  `NullPointerException` (no NPE — but a `ConcurrentModificationException`
  on iteration) is the only signal of a contract violation.
- `clearAll()` (lines 77-85) clears the three lists AND the two
  id-index maps. It does NOT call `rebuildIdIndexesLocked()` but
  because both `clear()` calls are explicit and the maps are
  already empty, the post-condition is identical. The pattern is
  inconsistent with `replaceAll` (which calls `rebuildIdIndexesLocked`
  after assigning the new lists). Make `clearAll` mirror
  `replaceAll` for clarity, or document the divergence.
- The `lock` field is private but exposed via `lock()` (line 49).
  A test that does `synchronized (state.lock())` (the documented
  way) still works, but every delegate uses `state.withLock(...)`
  and `state.withLockReturning(...)` which serialize correctly. The
  raw `lock()` accessor lets callers bypass the helper and acquire
  directly, which is fine for tests but is the exact pattern the
  Javadoc warns against. Make it package-private and reserve for
  the facade package.

**[src/domini/facade/ShelfDelegate.java]** — [severity: HIGH]
- `addLlista` / `deleteLlista` / `renameLlista` / `setLlistaColor`:
  persistence write first, then in-memory mutate. Unlike the book
  delegate, the in-memory mutate uses `llista.equals` (id-based) to
  remove the entry, so concurrent renames or deletes can be lost —
  e.g. two threads renaming the same shelf to different names will
  both succeed in DB (last writer wins) but the in-memory map
  will hold whichever entry the second mutator removed, leaving
  the DB out of sync with the in-memory state. Either serialize
  the in-memory mutate under the same lock that took the
  pre-decision (re-look-up the Llista by id after the DB write),
  or hold the lock across the DB call.
- `swapLlistesOrdreLocked` (lines 128-149) is documented to require
  the caller to hold the lock — true, but it then calls
  `cp.updateLlistaOrdre(...)` (DB) while still holding the lock.
  Same serialization concern as `addLlista` above; a slow DB
  freezes all other state operations. Use a copy-then-commit
  pattern: capture both `ordre` values, release the lock, run
  the two DB updates, reacquire and re-check.
- `moveLlistaUp` (line 96) and `moveLlistaDown` (line 102) do not
  validate that the second list (`i-1` / `i+1`) has a different
  id; if two consecutive shelves share an id (corrupted state),
  the swap becomes a no-op but the `updateLlistaOrdre` calls
  still fire. Add an `id != otherId` guard.

**[src/domini/facade/TagDelegate.java]** — [severity: MEDIUM]
- `getLlibresWithTag` / `getTagsForLlibre` / `removeLlibreFromTag` /
  `addLlibreToTag` (lines 66-77) are pure pass-throughs. This is
  fine for now, but the Javadoc claims delegates "follow the same
  contract" — there is no contract, just a delegated call. The
  methods are package-private; the facade exposes them via
  `ControladorDomini`. Document the intent (centralization point
  for future tag-side caching) or inline the pass-throughs at the
  facade.
- `addTag` (lines 35-46) and `deleteTag` (lines 48-55) also
  race between DB and in-memory state like `ShelfDelegate`. Two
  threads creating the same tag name: first passes the DB
  UNIQUE constraint, second throws — but the in-memory list ends
  up with one entry. Today the second thread never reaches the
  in-memory mutate, so it's correct, but if the DB ever drops
  the unique constraint (or the tagDao starts using `INSERT
  IGNORE`) the in-memory list will duplicate the entry.

**[src/domini/Llibre.java]** — [severity: MEDIUM]
- The class is a mutable 30+-field POJO with both setters and a
  Builder. The class Javadoc (lines 6-15) admits this and proposes
  composing `LlibreCore + extension state` as a future refactor.
  Today: the Builder is used only by `LlibreMapper` (DAO read path)
  and one test helper. Every controller in presentacio uses
  individual setters. The duplication is a maintenance trap:
  adding a 28th field requires touching 5 files. The Builder's
  `forInsert` / `forUpdate` separation in `LlibreFieldBindings`
  proves the value of declarative batching.
- Lines 32-34: `nomCa`, `nomEs`, `nomEn` are nullable. The setters
  at lines 116-120 call `nullIfBlankTrim` which is correct, but
  `getDisplayNom` (lines 128-134) only treats them as nullable
  when null; it does NOT distinguish empty string from null. With
  `nullIfBlankTrim` the empty-string case cannot occur, so the
  code is safe — but the API would be cleaner with `String getNomCaOrNull()`
  and `boolean hasNomCa()`.
- `bindUpdateableFields` (lines 52-64) sets every field including
  `autor` from the parameter. In the test path
  (`Presentacio.detalles.DetallesLlibrePanelControl.EditHandler`,
  presentacio/detalles/control/DetallesLlibrePanelControl.java:381),
  the caller parses `textAutor` into a `List<String> autors` and
  then calls `a.setAutors(autors)` which OVERWRITES the `autor`
  string. So `bindUpdateableFields(target, isbn, nom, autor, ...)`
  is called with the joined `autor` and then `setAutors(list)`
  overwrites it. The two-step order is correct today, but the
  implicit "setter overrides Builder" precedence is undocumented.

**[src/domini/Llista.java]** — [severity: MEDIUM]
- `equals` / `hashCode` (lines 58-65) compare by id. Two `Llista`
  with id=0 (in-memory, not yet persisted) are never equal — the
  Javadoc (lines 35-38) explains this for `Tag` but the same
  pattern is here. A transient `Llista` (id=0) used as a
  `HashMap` key will collide with every other transient `Llista`.
  Consider using a stable hash for transient (id=0) instances, or
  forbid transient `Llista` as a `HashMap` key.
- `setNom` (lines 35-38) throws `IllegalArgumentException` on
  blank, but the same check is duplicated in
  `domini.facade.ShelfDelegate.addLlista` (line 40) and
  `renameLlista` (line 62), and the message is hard-coded in
  English. Route through `BibliotecaException.Validation(I18n.t(...))`
  for i18n.

**[src/persistencia/ControladorPersistencia.java]** — [severity: MEDIUM]
- `synchronized` on every public method (lines 104-208). The
  class Javadoc (lines 27-29) explicitly proposes a
  `ReadWriteLock`. With 200+ entries the UI is mostly read-bound
  (table render, filter, search); under sustained write (e.g.
  bulk import + UI scrolling) every paint thread queues on the
  monitor. Convert reads to `ReentrantReadWriteLock.readLock()`
  and writes to `writeLock()`. The TODO is older than the
  facade split; now is the time.
- `getAllLlibres()` and `getAllLlibresSummary()` (lines 104, 111)
  are the same implementation. The Javadoc on
  `interficie.BookReader.getAllLlibresSummary` (interficie/BookReader.java:34)
  documents the contract, but the body is identical. The summary
  contract lets the in-memory test stub skip work; the production
  path always returns the full row. Either implement the summary
  contract here (omit `descripcio`/`notes` from the SELECT) or
  delete the alias.
- `setDbProfile` (via `setDbType` line 296) writes
  `dbHost = "localhost"` and `dbUser = "user"` whenever type is
  H2. After a MariaDB session, switching back to H2 overwrites
  the previous host/user with defaults — losing any non-default
  MariaDB values that the user may want to keep for later
  reconnection. Document this destructive fallback or preserve
  the previous values.

**[src/persistencia/ServerConect.java]** — [severity: HIGH]
- `createDatabase(ConnectionConfig)` (lines 155-181) catches
  `Exception` and re-throws as `RuntimeException` with the
  i18n-flavoured message `I18n.t("err_db_init_" + cfg.dbType())`.
  If `dbType` is neither `h2` nor `mariadb` (e.g. a typo or a new
  type) the key lookup falls through to `err_db_init_h2` (because
  `I18n.t` returns the key when missing) — so the user sees
  the wrong error message. Use a switch on `dbType()` with an
  explicit else that throws `IllegalArgumentException("Unknown
  dbType: " + cfg.dbType())`.
- `MIGRATIONS` (lines 37-111) is a 75-entry array. The MIGRATION
  32-34 sequence (lines 95-105) is fragile: 32 inserts into
  `autor`, 33 inserts into `llibre_autor`, 34 drops the `autor`
  column. If 33 fails, 34 still runs and the legacy column is
  gone — but 33 left rows partially migrated. The "skip on
  unmigrated rows" guard in `runMigration34DropAutor` is correct
  for the all-or-nothing 33 path, but 32 and 33 are NOT
  wrapped — if 32 fails (e.g. H2 schema conflict), 33 will still
  attempt, with no transaction. Wrap the 32+33+34 sequence in a
  single H2 transaction for the H2 branch; for MariaDB, document
  the operator-recovery steps.
- `columnExists` (lines 310-320) does two `meta.getColumns` lookups
  (original-case and upper-case) but ignores the result of the
  first one if the second is run regardless. Re-read the Javadoc
  to confirm: line 314 says "if (rs.next()) return true;" — so the
  second lookup only runs if the first returns false. OK, this
  is correct; but the second lookup always returns at the
  post-uppercase table/column names, which on a case-sensitive
  DB might miss a column whose only the lowercase form exists.
  `DatabaseMetaData.storesLowerCaseIdentifiers` would be the
  correct way to choose.
- `loadDriverFromLib` (lines 338-351) builds a `URLClassLoader`
  per call. Each instance pins the JAR in memory; the GC root
  is the static `Driver` field returned, but the
  `URLClassLoader` itself becomes unreachable. The H2 and MariaDB
  drivers cache static state, so this is fine in practice, but
  on driver upgrade (loading a new jar) the old
  `URLClassLoader` is never closed. Not a leak for a desktop app
  but worth documenting.
- `runMigrationH2` (lines 246-270) saves and restores
  `prevAutoCommit` for each migration. The outer `runMigrations`
  caller (lines 218-239) also saves and restores in a
  `try/finally` (line 237). The two saves race if the outer
  `prevAutoCommit` was already `false` (e.g. another caller
  changed it) — the inner save captures `false`, the inner
  restore sets it to `false`, the outer `finally` restores to
  the original `true` — net effect is correct, but the
  bookkeeping is fragile. Document the contract or remove the
  inner save/restore.

**[src/persistencia/LlibreDaoCore.java]** — [severity: HIGH]
- `executeSQLFile` (lines 173-192) reads the file via
  `Files.readAllBytes` (line 177) and parses the bytes back into
  a String. For a 50 MB backup file this allocates a 50 MB
  `byte[]` and a 50 MB `String` (with the UTF-8 decoder holding
  a CharBuffer of similar size). For multi-MB backups the peak
  memory is 4× the file size. Stream the file instead: read
  line by line with a `BufferedReader`, accumulate the
  multi-line statement, dispatch to the DB. Also avoid the
  `withTransaction` wrapper for the file-level
  BEGIN/COMMIT/INSERT statements, since each statement is its
  own auto-commit operation on MariaDB (the wrapper is a no-op
  there).
- `insert` and `update` (lines 90-127) wrap the column-by-column
  bind in `LlibreFieldBindings.forInsert` /
  `LlibreFieldBindings.forUpdate`. The wrapper class
  `LlibreFieldBindings` is 142 lines; the centralization is
  good, but `forInsert` and `forUpdate` are two near-identical
  Object[] builders. The difference is `imatge_blob` is in
  INSERT but not UPDATE, and the field order is shifted. A
  single helper that takes a `boolean includeBlob` parameter
  would halve the maintenance surface.
- The `LLIBRE_COLUMNS`, `LLIBRE_COLUMNS_L`, and
  `LLIBRE_COLUMNS_L_LIGHT` constants (lines 25-49) are three
  near-duplicate strings. Each is a different hand-written
  projection of the same 28 columns. Adding a new column
  requires editing all three (plus `LlibreFieldBindings`,
  `LlibreMapper.fillLlibreTail`, the schema migration list,
  the backup `writeLlibreINSERT` printf, and the
  `GoodreadsExportService` row builder). Consider a single
  `static String buildColumnList(String prefix, boolean
  light)` that emits the column list from a shared
  `List<ColumnSpec>` record array. The 3 lists would
  collapse to one source of truth.

**[src/persistencia/LlibreSearchDao.java]** — [severity: MEDIUM]
- `search` (lines 26-70) builds the SQL via StringBuilder with
  20+ `if (f.getXxx() != null) sql.append(...)` branches. The
  pattern is identical for every condition. Extract a helper
  `addCondition(StringBuilder sql, List<Object> params, String
  clause, Object value)` that no-ops when value is null. The
  method shrinks from 45 lines to 20 and the place-counting
  bugs (`?` placement) become one-line.
- `bindParam` (lines 72-82) uses pattern-matching `switch` for
  the boxed primitives but `case null` sets `Types.NULL`. Today
  the SQL builder never inserts a literal `null` (every
  condition is guarded by `!= null`), so the `case null` is
  dead. Keep the defensive fallback but log at FINE level when
  it fires so dead code becomes visible if a future call site
  changes.
- `LlibreFilter.getLlistaId()` (line 31) is documented to take
  precedence, but the SQL has `JOIN llibre_llista ll ON
  l.ISBN = ll.isbn AND ll.llista_id = ?` and `JOIN llibre_tag
  lt ON l.ISBN = lt.isbn AND lt.tag_id = ?` as two separate
  branches. When both `llistaId` and `tagId` are set, the
  result is the intersection (rows that are in BOTH), but
  applying them independently as separate `AND` clauses in
  the WHERE would also be the intersection. The current
  approach is correct, but the join is on the same alias `l`
  — there's a subtle bug if MariaDB complains about a duplicate
  join alias in the rendered SQL. Test with both filters set.

**[src/persistencia/TagDao.java]** — [severity: MEDIUM]
- `AUTOCOMPLETE_COLUMNS` (lines 21-22) is the SQL whitelist for
  `getDistinctValues`. The matching in-memory extractors live
  in `domini.facade.StatsDelegate.IN_MEMORY_EXTRACTORS` (line 32
  of that file). The two lists must be kept in sync manually;
  nothing in the code enforces it. Add a startup assertion or
  build the in-memory list by reflecting on the SQL whitelist.
- `llibreTagCache` (line 24) is `volatile` and the read path
  (line 124) is `if (llibreTagCache != null) return
  llibreTagCache;`. The `getAllLlibreTag` method holds the
  instance monitor (line 123 `synchronized`); the read
  path does not. A `setRow` operation on the volatile is
  atomic, but the read can see a partially-published
  `List.copyOf` reference. Actually `List.copyOf` is
  published via a single store — the volatile semantics

**[src/persistencia/Schema.java]** — [severity: LOW]
- `CLEAR_ORDER` (lines 16-19) is a 9-element array. The

## Calibration

1. **Weakest finding included.** The
   `EstadoCOnfigurationDialog` / `ConfiguracioDataSection.btnPerfils`
   profile-name sanitization (see `ConfiguracioDataSection.java:101`)
   was flagged as LOW but the evidence is thin: I noticed
   `chosen.trim().replaceAll("[^a-zA-Z0-9_-]", "_")` strips non-ASCII
   characters from a user-typed profile name silently, but I did
   not test whether the resulting profile name actually
   round-trips through `H2` or breaks a re-connection flow. To
   strengthen this, I would write a JUnit test that types
   "café" in the profile field, clicks save, then verifies
   that `Config.getDbProfile()` returns "caf_", and that
   re-opening the dialog shows "caf_" (not "café"). The
   concrete failure mode (silent rewriting) is plausible but
   unconfirmed; the fix is trivial (add a warning dialog).
2. **New class of issue to look for next run.**
   I missed auditing the **SQLite-on-Calibre** integration
   end-to-end. The shell-out in `BookImporter.importCalibre`
   passes a multi-MB SQL through `sqlite3` stdout and
   parses with `line.split("\\|", -1)` — but I did not
   test with a 1000-book Calibre library to see whether
   the process timeout (30s, line 99) is sufficient, nor
   did I check the charset handling for non-UTF8 Calibre
   libraries. The pipe-delimited SQL also assumes no book
   title contains a literal `|` character, which is rare
   but possible. A next-run test fixture with a Calibre
   library containing `|`-titled books would expose the
   parser's blind spot.
3. **Confidence score.** `confidence: 72`
   The HIGH-severity findings (BookDelegate race, BackupDelegate
   EDT loadHeavyFields, LlibreSearchDao SQL builder, OpenLibrary
   regex fallback, BookExporter HTML inline covers, FilterController
   SwingWorker, DetallesLlibrePanelControl perm-disabled ISBN) are
   all backed by code references and reproducible. I am most
   unsure about the Llibre/LlibreFilter maintainability trap
   and the BackupService auto-backup prune race — both are
   plausible but require runtime evidence (long-running test
   with concurrent restore + auto-backup) to be certain. I would
   not bet money on every LOW-severity nitpick being a true
