────────────────────────────────────────────────────────────────────────────────
  R1 — FilterDrawerPanel + FilterController: declarative table-driven
       field registry to remove parallel-field boilerplate
────────────────────────────────────────────────────────────────────────────────

WHY:    FilterDrawerPanel.java:28-66 declares 32 JTextField/JCheckBox/
        JComboBox/JButton fields one-per-line, and buildFilterDrawer()
        (lines 72-385) instantiates them in the same order with the same
        field-naming convention, then wires them in FilterController.java
        :33-60 listener-by-listener. The hand-written 384-line panel has
        grown organically across many sessions and is the most likely
        place a new filter field will be forgotten (wiring, debounce, or
        persistence). The same pattern recurs in DetallesLlibrePanel
        (separate file) and is the reason those 430-line detail panels
        exist.

GOAL:   Replace the imperative `private JTextField textISBN; ...` block
        with a single declarative table (List<FilterFieldSpec>) that
        drives (a) field construction, (b) GroupLayout positioning,
        (c) action-listener wiring, (d) filter-spec extraction.
        End state: adding a new filter is a single-line table entry.

FILES TO TOUCH:
  src/presentacio/FilterDrawerPanel.java     refactor ~30 fields → table
  src/presentacio/FilterController.java      use registry to wire listeners
  src/presentacio/FormFieldRegistry.java     NEW — already exists as
                                              skeleton (see §6 finding 2)
  test/presentacio/FilterDrawerPanelTest.java
                                             NEW — verify field count,
                                              wiring, debounce, apply/clear

APPROACH (4 steps, each independently mergeable):
  1.  Extract `record FilterField(String key, String label, FilterKind
      kind, Object defValue)` and a `Map<String,JComponent> components`
      registry. Keep the public getters that callers already use
      (`getTextISBN()`, `getTextNom()`, …) so nothing outside the
      drawer/control changes. Add a build-time registry: for each
      spec, instantiate the right JComponent subclass, store in
      `components.put(key, c)`, and replace field reads with
      `lookup(key)`.
  2.  Replace the 32 individual field declarations + per-field
      setters/getters with one getter per legacy call site that does
      `return (JTextField) components.get(key)`. This keeps
      FilterController and MainFrameControl compiling unchanged.
  3.  Generate GroupLayout from the spec list. One method:
      `GroupLayout.SequentialGroup vertical(GroupLayout gl,
      List<FilterField> specs)` that loops and adds the label+control
      pair. No more hand-rolled parallel-group chains.
  4.  Replace FilterController's 12-line `addActionListener` chain
      with `specs.stream().filter(s -> s.kind == TEXT)
      .forEach(s -> lookup(s.key).addActionListener(enterFiltrar))`.

RISKS:
  -   32 getters are still public API; if any external class reads
      them (MainFrameControl, StressTest, KeyboardShortcuts), the
      find-replace has to be exhaustive. Mitigation: keep the
      getters, just change their body to a map lookup.
  -   Layouts with grouped rows (3 numeric-range rows, 2 read
      checkboxes) need a `Row` wrapper. Mitigation: the FilterRow
      + FilterRange records from C5 are the exact pattern to reuse
      — the C5 split was the precondition for this refactor.
  -   Theme color application (UITheme.palette().bgField(),
      etc.) is scattered through buildFilterDrawer. Mitigation: a
      single `applyTheme(Component c)` pass that walks the registry
      after build.

VERIFY:
  scripts\test.bat   (the existing FilterController is covered by
                      the per-class tests for MostrarBibliotecaPanel;
                      add FilterDrawerPanelTest to lock the public
                      getter returns)
