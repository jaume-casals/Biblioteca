[ST-01] ISBN COLLISION — uniqueness not guaranteed across runs or within run
  File: StressTest.java:33, 1363-1365
  Code: isbnCounter = System.currentTimeMillis() % 900_000_000L;
        uniqueISBN() { return 9780000000000L + (isbnCounter++ % 999_999_990L); }
  Problem: isbnCounter is static and seeded from currentTimeMillis. If two runs start
           in the same millisecond (or share a persistent DB), collisions are possible.
           The modulo 999_999_990 is irrelevant here (Long can hold 9_999_999_900 values
           before overflow — far beyond any test run size, so the modulo is dead code).
           But the time-based seed is the real problem: consecutive runs with the same
           millisecond timestamp get identical sequences.
  Fix: Use AtomicLong with a random base + monotonic increment, no time seed:
         private static final AtomicLong ISBN_BASE = new AtomicLong(
             9780000000000L + (new Random().nextLong() & 0x7FFFFFFFFFFFFFFFL) % 900_000_000L);
         private static long uniqueISBN() { return ISBN_BASE.addAndGet(1); }
  Fix: Also assert uniqueness before each create (addTag already does this).

[I18N-01] CSV parser: split on "," — fields containing commas are silently broken
  File: I18nAudit.java:32-33
  Code: String[] row = s.split(",", 4); if (row.length >= 4) out.put(row[0], new String[]{row[1],row[2],row[3]});
  Problem: If a CSV field value contains a comma (e.g., "New, Improved" as a translation),
           split(",", 4) would produce more than 4 fields and silently discard the value.
           More importantly, row.length is checked as >= 4, but if the split produces
           exactly 4 parts and one of the translations contains a comma, the last part
           gets truncated. For I18n keys, translations should be quoted or the parser
           should use a proper CSV library.
  Fix: Use a proper CSV parser (e.g., opencsv) or handle quoted fields:
         String[] row = s.split(",(?=(?:[^\"]*\"[^\"]*\")*[^\"]*$)", -1);
         This regex split handles quoted fields with embedded commas.

[I18N-02] auditHardcodedCatalan: only warns on first match per file — misses all others

  Problem: Pattern.matcher.find() finds only the FIRST match in the file. If a file has

[I18N-04] auditDuplicateKeys: does not detect duplicates within the same file
  File: I18nAudit.java:100-116
  Code: firstFile.putIfAbsent(key, p.getFileName()); if (prev != null && !prev.equals(...))
  Problem: firstFile is keyed by key only (not by filename). If the same key appears twice
           in file1.csv, the second occurrence overwrites the first in firstFile but the
           check prev != null && !prev.equals(...) would only trigger if the same key
           appears in a DIFFERENT file than the first. So duplicates within a single
           file are not detected.
  Fix: The logic is: firstFile maps key -> first filename where seen. If we see the same
        key in a different file, that's a duplicate. But if the same key appears twice

         Map<String, Map<String, Path>> keyToFileToPath...
