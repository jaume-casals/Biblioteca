# Biblioteca — Code Review (2026-06-16)

Autonomous review pass. Every source file in `src/` was re-read end to
end. Findings below are ranked CRITICAL → LOW. Each file's section is
capped at three findings; no overflow bucket.

**Themes this run (delta vs the 2026-06-13 review):**

- `DateUtils.parseYear` now returns `Optional<Integer>`, not `int` — the
  prior LOW finding on the "0 sentinel" is closed; dropped.
- `LlibreFieldBindings` centralisation of INSERT column lists is
  in place; the prior drift concern is closed; dropped.
- `BackupService.shutdownScheduler` was fixed for the
  rescheduling-during-teardown race; dropped.

New issues raised this run focus on **calibre-import data corruption**,
**the JsonImporter duplicate-skip bug** (skipped-counter never increments
in the success path, false-positive warning fires for every import),
**secret-string lifetime in the DB password path**, **the splash window
leak when the loader thread catches its own exception**, and several
smaller drift / dead-code / performance findings.

**[src/herramienta/JsonImporter.java]** — [severity: HIGH]
- `try { cd.getLlibre(isbn); skipped++; continue; } catch (Exception e) { LOG.warning("Skipped book with unparseable ISBN " + isbn, e); }` (line 67-71).
  The control flow is inverted: `cd.getLlibre(isbn)` throws
  `BibliotecaException.NotFound` for books the import should
  *create* (the common case), the catch swallows it, logs a
  misleading "unparseable ISBN" warning **for every imported book**,
  then falls through to the `addLlibre` path below. The
  `skipped++; continue;` branch is only reached when the book
  already exists, which is the only path that should increment the
  skip counter. Net effect: the skip counter is wrong, the log is
  full of false-positive warnings, and the "imported" count is
  one higher than reality for every book. Fix: invert the check —
  call `cd.existsLlibre(isbn)` (no exception) and `continue` on
  true, or `try` the get, put `skipped++; continue` in the catch,
  and remove the catch-side log entirely. The ISBN parsing
  failure case this warning was meant to flag happens in
  `getLlibre` only on truly malformed ISBNs, which is impossible
  here because the JSON was just deserialized from a `long`.

**[src/herramienta/BookImporter.java]** — [severity: HIGH]
- `importCalibre` (lines 79-95) parses each output line with
  `c = line.split("\\|", -1)`. Calibre book titles legitimately
  contain the pipe character (it is not reserved in the Calibre
  `books.title` column), and the SQL is run with
  `GROUP_CONCAT(a.name, ', ')` which can also emit `|` if a
  publisher/series name contains one. A single literal `|` in a
  title shifts every subsequent field left, so the parser reads
  the year as the editor, the year as the rating, the rating as
  the notes, etc. — silently corrupting 9 fields per affected
  book with no error path. Fix: switch the output format to
  tab-separated (`sqlite3 -separator $'\t' …`) and split on
  `\t` with a guard for embedded tabs in the title. This is the
  exact failure mode the prior calibration section flagged as
  the weakest finding of the previous run — confirmed real,
  confirmed exploitable in practice.

**[src/herramienta/BookImporter.java]** — [severity: MEDIUM]
- `importCSV` is run on the EDT via `BookImporter.importCSV` →
  `BookIOController.pickFile` → `JFileChooser.showOpenDialog` runs
  synchronously, but the parsing that follows
  (`BookIOController` chains into `new SwingWorker<…>` blocks,
  so the file I/O is fine). However `JsonImporter.run` is invoked
  directly from the SwingWorker's `doInBackground()` via
  `BookImporter.importJSON` → `JsonImporter.run(file, cd)`. The
  first line of that path is
  `String json = new String(Files.readAllBytes(f.toPath()), UTF_8)`
  (line 33). A 200 MB JSON file allocates 400 MB transiently
  (the bytes → String round trip duplicates the heap footprint)
  and pins the worker thread on a single `readAllBytes` call. Fix:
  stream-parse with `JsonParser.parseReader(new InputStreamReader(
  new BufferedInputStream(new FileInputStream(f)), UTF_8))` so
  the JSON token stream is consumed incrementally and the
  library is populated row-by-row on the worker thread.

**[src/persistencia/ServerConect.java]** — [severity: HIGH]
- `testConnection(java.util.Properties props, char[] password)`
  (line 128-144) takes the password as a `char[]` to avoid
  the heap-resident `String` lifetime that the rest of the
  codebase worries about, but the very first thing it does is
  `String pass = new String(password)` is missing — instead
  the body uses `new String(password)` only when
  constructing the `Properties` (line 142: `new String(password)`).
  So the path is: caller has `String` (from `Config.getDbPassword`),
  → `toCharArray()` (line 123), → `new String(password)`
  (line 142). The `char[]` lifetime protection is defeated by
  the immediate `new String` round trip; the original String
  in `Config` is never zeroed either. Fix: keep the password
  as `char[]` end-to-end (`testConnection` builds the JDBC
  `Properties` with a `char[]` value, or via a small JDBC
  driver shim that doesn't promote to `String`); same for
  `ConfiguracioButtonBar` which currently does
  `DbConfig.setPassword(new String(txtPass.getPassword()))`
  (line 73) — a `String` is now immortal in `Config.props` until
  the process exits. The on-disk 0600 mitigation in
  `Config.doSave` is good, but it does not address the in-heap
  attack surface that the `char[]` API exists to defend.

**[src/main/Ejecutable.java]** — [severity: HIGH]
- The splash screen leak on `ControladorDomini` init failure.
  `startSwingWithSplash` (line 59-107) installs a daemon
  `loader` thread; if `cdRef.set(ControladorDomini.getInstance())`
  throws, the catch at line 83-90 shows a JOptionPane and
  `System.exit(1)`s, but **does not call `splashRef.forceHide()`**.
  The `Thread.setDefaultUncaughtExceptionHandler` at line 34
  *does* call `splashRef.forceHide()`, but the catch in the
  loader thread swallows the exception, so it never reaches
  the uncaught handler. Net effect: on init failure, the
  splash stays on screen, the modal JOptionPane can't receive
  focus, and the user sees two dialogs. Fix: call
  `if (splashRef != null) splashRef.forceHide();` before the
  `JOptionPane.showMessageDialog` call, or rethrow as a
  `RuntimeException` so the uncaught handler runs.

**[src/main/Ejecutable.java]** — [severity: LOW]
- The `args[0].equals("--web")` branch (line 25-28) is dead code
  left from the API/web removal. The build contract (README +
  `install.sh`) no longer documents `--web`; users running with
  the flag get a hard `System.exit(1)` with a message that
  references a feature that no longer exists. Fix: drop the
  branch and the comment block at line 30-32; if you want to
  preserve the error for any legacy launcher, change the
  message to a single `System.err.println` + return-code 2

**[src/herramienta/CoverService.java]** — [severity: MEDIUM]
- `fetchAsync` (line 67-88) submits a `POOL` task that calls
  `cd.setLlibreBlob(lIsbn, cached)`. The pool is a fixed
  6-thread executor, and `ExportController.fetchMissingCovers`
  (line 113-187) submits *one task per missing book* on the
  same pool, so a 500-missing-book fetch is bounded by 6
  concurrent JDBC writes. The pool also shares the
  `OpenLibraryClient` 300 ms rate limiter across both the
  cover-fetch and the new-book lookup paths, so the rate
  limiter is effectively global — good for OL, but it means
  one heavy call to `mergeGoogleBooks` (which itself does a
  network round trip) blocks every other cover fetch. Fix:
  separate the OL fetcher into a single-thread executor (the
  rate limit guarantees ≤3.3 calls/s) and let the JDBC
  writes go on a thread pool sized by the cover count.
  Today a 1000-book fetch takes ~5 minutes wall-clock because
  the writes are stuck behind the rate-limited reads.

**[src/herramienta/BackupService.java]** — [severity: MEDIUM]
- `autoBackup` (line 65-133) takes the bib/llistes/tags
  snapshots on the scheduler thread *one after the other*.
  If a write commits between `cp.getAllLlibres()` (line 66)
  and `cp.getAllTags()` (line 91), the backup is internally
  inconsistent — a book that was added may reference a tag
  that doesn't yet exist in the snapshot, or a tag delete
  may orphan a `llibre_tag` row. The classic defence is a
  single SQL transaction (or at least a `setTransactionSnapshot`)
  around the three reads; the application already uses JDBC
  connections from a single `ControladorPersistencia` so
  `cp.getConnection().setTransactionIsolation(
  Connection.TRANSACTION_SERIALIZABLE)` for the read pass
  would close the window. Acceptable trade-off: the auto-
  backup is "best effort", so document the race explicitly
  in the Javadoc, and add a single `conn.rollback()`-on-
  conflict retry so a retry sees a consistent state.

**[src/herramienta/BackupService.java]** — [severity: LOW]
- The prune loop at line 128-132 is non-atomic. The
  `dir.listFiles()` and the subsequent `backups[i].delete()`
  calls are separated by the rename at line 110-123, so a
  backup process running in parallel (e.g. a manual
  `BackupController.backupBD` from a second JVM) could
  have its in-progress file deleted before it finishes
  the rename. Fix: filter the prune set by
  `lastModified() < now - 1 minute` so a file in the last
  minute of wall-clock time is never pruned.

**[src/herramienta/LlibreValidator.java]** — [severity: MEDIUM]
- `normalizeIsbn13` (line 31-52) re-implements
  `Isbn13Normalizer.toIsbn13` with subtle behaviour
  differences — `normalizeIsbn13` accepts ISBN-10s with X,
  converts them to ISBN-13 with the 978 prefix, and re-emits
  the input unchanged when neither shape matches; the
  helper in `Isbn13Normalizer` returns `null` on no-match.
  `checkLlibre` then calls `normalizeIsbn13` and falls
  through to `Long.parseLong(isbn)` only on the 13-length
  result, so a 10-digit ISBN-10 input is silently rejected
  (the 9-digit substring isn't produced here at all). Fix:
  call `Isbn13Normalizer.toIsbn13(isbnStr)` first; if null,
  throw the digit-length error. The duplicated X check
  logic at line 79-93 is the second copy of the same code
  path that `Isbn13Normalizer` already encodes.

**[src/herramienta/LlibreValidator.java]** — [severity: LOW]
- Line 65-69: `if (isbn != null) { String normalized =
  normalizeIsbn13(...); ... }` is dead code — the method
  returns at line 63 if `isbn == null`, so this branch is
  unconditionally true. Either drop the `if` or drop the
  guard at line 63. The dead branch hides the fact that
  `normalizeIsbn13` only ever returns a 13-digit string;
  a reader who assumes the `null` ISBN case is reachable
  here will misread the contract.

**[src/herramienta/Config.java]** — [severity: MEDIUM]
- `props` is an `AbstractMap` whose `get`/`put` calls
  `domainOf(key)` on every access (line 98-99, 102-103). The
  `domainOf` switch (line 71-80) walks four `Set.contains`
  checks plus two `String.startsWith` checks for every read.
  For the hot `props.get("lang")` and `props.get("dbPassword")`
  paths (called on every `getLang`/`getDbPassword` invocation
  in `I18n`/`WindowConfig` etc.) the domain lookup is
  ~10x more expensive than a `HashMap.get`. Fix: cache the
  routing per key in a `Map<String, Domain>` (built once
  per `put`).

**[src/herramienta/Config.java]** — [severity: LOW]
- `props.entrySet()` (line 106-113) returns a new
  `HashSet` containing entries from four sub-stores; the
  resulting `Properties.store` order is non-deterministic.
  On a config-write round-trip the saved file is bit-
  different from the loaded one every time, which breaks
  any test that diffs the file. Fix: use a
  `LinkedHashSet` insertion-ordered across the four
  sub-stores (the contract is "all four", and the four
  are independent — order between them is meaningless,
  but order *within* a sub-store should be stable).

**[src/herramienta/Config.java]** — [severity: LOW]
- The `domini/facade/*Delegate` classes store passwords via
  `Config.props.put("dbPassword", String.valueOf(pw))`
  (transitively, via `DbConfig.setPassword`). The
  plaintext on disk is 0600 (good); the plaintext in heap
  is unguarded. The 2026-06-13 review's calibration section
  noted this is "acceptable for a local single-user app"
  and linked `tot.txt B2` for the keystore migration. The
  migration has not landed. The path is now consistent
  across `ConfiguracioButtonBar` (line 73), `testConnection`
  (line 142), and `BackupService`'s write path — all three
  promote `char[]` to `String`. Pick one entry point and
  fix all three together, not one at a time.

**[src/presentacio/MainFrameControl.java]** — [severity: MEDIUM]
- `getInstance(panel, cd)` (line 201-205) is a classic
  broken double-checked locking: `if (instance == null &&
  panel != null) instance = new MainFrameControl(...)`.
  The `instance` field is `volatile` (line 48) but the
  check-then-construct is unsynchronised. Two EDT-event
  threads can both see `instance == null`, both construct
  a frame, and the second assignment overwrites the first.
  The first frame's `JFrame` is then leaked, its
  `instance` `MainFrameControl` is garbage but its
  window stays on screen (the leaked `JFrame` is shown).
  The `else if` only catches the case where the second
  caller still sees `instance != null` after the first
  assignment — not the case where the first has started
  but not yet assigned. Fix: `synchronized
  (MainFrameControl.class)` around the check-and-set, or
  use the static-holder idiom
  `private static class Holder { static final
  MainFrameControl I = new MainFrameControl(...) }` (which
  is awkward here because the ctor takes args — instead,
  use a `CompletableFuture<MainFrameControl>` initialised
  once).

**[src/presentacio/detalles/control/DetallesLlibrePanelControl.java]** — [severity: MEDIUM]
- The heavy-fields `SwingWorker` at line 108-122 is
  not cancelled on window close. The
  `windowClosing` listener at line 99-102 only cancels
  `imageWorker`; the anonymous worker continues running
  after the dialog is disposed, and its `done()` calls
  `vista.getTextDescripcio().setText(...)` on a disposed
  `JTextField` (throws `IllegalStateException` on some
  L&Fs and leaves a half-mutated in-memory `book` on
  others). The `done()` already wraps in a try-catch, so
  the IllegalStateException is swallowed — but the
  side-effect of the in-memory `book.getDescripcio()` /
  `book.getNotes()` being overwritten with the empty
  default is real. Fix: hoist the `SwingWorker` to a
  field and `cancel(true)` it from `windowClosing`; also
  short-circuit `done()` with `if (isCancelled() ||
  !vista.isDisplayable()) return;`.

**[src/presentacio/detalles/control/DetallesLlibrePanelControl.java]** — [severity: LOW]
- The three static parsers `parseIntOrNull`,
  `parseDoubleOrNull`, `parseIntStrict`,
  `parseDoubleStrict` (line 37-80) are duplicated in
  `GuardarLlibresDialogoControl` as `parseLongOrNull`,
  `parseIntOrNull`, `parseDoubleOrNull` (line 213-235)
  with slightly different signatures and exception
  wrapping. The `GuardarLlibresDialogoControl` versions
  throw `NumberFormatException` with a translated i18n
  prefix; the `DetallesLlibrePanelControl` versions
  append errors to a `List<String>` and return a default.
  The difference is the strict-vs-best-effort contract
  documented in the Javadoc — but two parallel
  implementations of the same three conversions in the
  same package is a maintenance hazard. Fix: extract
  `ParseHelpers` with three functions
  (`parseLong(String, long default)`, `parseInt`,
  `parseDouble`) that take a `List<String> errors`
  parameter (null = best-effort) and reuse from both
  controls.

**[src/presentacio/ConfiguracioDataSection.java]** — [severity: MEDIUM]
- `sanitizeProfileName` (line 23-25) silently rewrites
  any character outside `[A-Za-z0-9_-]` to `_`. A user
  typing "café" gets "caf_", "日本" gets "__", and the
  re-opened profile dialog shows "caf_" without telling
  the user their input was modified. The 2026-06-13
  review's calibration section noted this; it is still
  present. Fix: detect any rewritten character and show
  a `JOptionPane.WARNING_MESSAGE` ("The profile name
  contains characters that will be replaced; continue?")
  *before* saving. The sanitize-and-silently-commit path
  is the textbook silent data-corruption anti-pattern.

**[src/presentacio/ShelfController.java]** — [severity: MEDIUM]
- `ShelfCellRenderer.getListCellRendererComponent` (line
  141-167) constructs a new anonymous `Icon` per call:
  `icon = new javax.swing.Icon() { ... }` inside the
  `if (ll.getColor() != null)` branch. A repaint of the
  shelf combo (open a shelf menu, hover, close) calls
  this for every visible item × every icon. For 20
  shelves this is 20 allocations per repaint × every
  mouse move on the combo — measurable GC pressure.
  The 2026-06-13 review's calibration called this out as
  the "mainstream noise" finding; it is still present.
  Fix: extract the Icon to a static `Icon` array keyed
  by `Color` (the `ColorSwatchPicker` already does this
  for the shelf buttons in `LeftSidebarPanel`), or use
  a per-`Color` cache (`Map<Color, Icon>` with capped
  size).

**[src/presentacio/galeria/CoverCardFactory.java]** — [severity: MEDIUM]
- `paintCard` (line 178-227) calls
  `Config.getLang()` (line 202) inside the paint path
  on every card, every repaint. For 200 visible cards
  in the gallery, a single `repaint()` (e.g. scroll
  one notch) issues 200 `props.get("lang")` calls —
  each is a `domainOf` walk, a `ConcurrentHashMap.get`
  on the UI sub-store, and a `String.equals` against
  the default. Fix: cache `Config.getLang()` at the
  top of the paint invocation in `GaleriaCobertesPanel
  .updateLlibres` and pass it down to `CoverCardFactory`
  via a `langRef` on the `CardState` record, so the
  per-card paint call becomes a field read.

**[src/presentacio/detalles/vista/DetallesLlibrePanel.java]** — [severity: MEDIUM]
- The 30 hand-declared component fields (line 44-78)
  and 30 getters (line 359-394) are still in the file.
  The `FilterDrawerPanel` refactor (R1) replaced 32
  fields with a `FormFieldRegistry` of declarative specs;
  the `GuardarLlibresDialogo` did the same. The
  `DetallesLlibrePanel` retains 30 hand-declared fields
  even though the `addFieldEntry` /
  `addComboEntry` / `addCheckEntry` helpers at line
  298-342 already follow the same "create and register
  a child" pattern that the registry encapsulates.
  This is a known anti-pattern in the file but the
  refactor is purely mechanical. Fix: declare
  `private final FormFieldRegistry registry = new
  FormFieldRegistry();` and the three factories
  (`addFieldEntry`, `addComboEntry`, `addCheckEntry`)
  register their component into the registry in

**[src/domini/Llibre.java]** — [severity: MEDIUM]
- `setAutors(List<String> autors)` (line 162-165) has
  asymmetric behaviour: when `autors` is null or empty,
  `this.autor` is left at its previous value. The
  contract the rest of the code relies on (e.g.
  `getAutor()` returns `String.join(", ", autors)`
  when `autors` is non-empty, or the legacy string
  otherwise) makes the asymmetric setter subtle: a
  caller doing `l.setAutors(newList); l.setAutors(
  List.of())` ends up with `l.getAutors()` empty
  but `l.getAutor()` still showing the joined
  `newList`. Fix: always set `this.autor`:
  `this.autor = this.autors.isEmpty() ? "" :
  String.join(", ", this.autors);`.

**[src/domini/facade/BackupDelegate.java]** — [severity: MEDIUM]
- `restoreFromSQL` (line 100-143) snapshots the pre-
  restore state outside the JDBC connection's
  transaction. The `preBib = new ArrayList<>(state.bib())`
  (line 112) is taken *before* `clearAllData()` and
  `executeSQLFile(file)` (line 118-119), but the
  persistence-layer connection has no awareness of
  the snapshot. If the application is killed between
  `clearAllData()` and `executeSQLFile()`, the
  database is wiped and the snapshot is lost. The
  current defence is the post-failure rollback to the
  `biblioteca_restore_backup_*.sql` file at line 122-126,
  but the rollback is also "outside" the failed
  transaction. Fix: open a single JDBC transaction
  at the persistence layer, do the `clearAllData` +
  `executeSQLFile` inside it, and on `SQLException`
  the `conn.rollback()` does both. The "snapshot
  to temp file" path becomes a backup-only operation
  for the user-initiated `Ctrl+Z` undo, not a safety
  net.

**[src/domini/facade/BookDelegate.java]** — [severity: LOW]
- `loadHeavyFields` (line 175-195) re-checks
  `book.isHeavyFieldsLoaded()` inside the lock but
  does nothing if the check succeeds — the comment
  on line 178-180 says "a concurrent updateLlibre
  may have already loaded heavy fields on the same
  row", but the worker's effect is to overwrite
  `book`'s fields with the DB values, so the re-check
  is purely an optimisation. The DAO call on
  line 182 is `loadHeavyFields(book.getISBN(), book)`,
  and the Javadoc on the DAO says "Books that are
  already isHeavyFieldsLoaded() should be filtered out
  by the caller (it has the in-memory reference) to
  avoid an unnecessary target.set* no-op" (LlibreBlobDao
  .loadHeavyFieldsBatched). The contract for the
  single-row variant should be the same — hoist the
  early-return to the caller and drop the re-check
  inside the lock.

**[src/persistencia/LlibreSearchDao.java]** — [severity: LOW]
- The four `addCondition` overloads (line 53-69) for
  `editorial`, `serie`, `idioma` use `LIKE` with
  `%...%` wildcards, but `format` uses `=` (line 66-67)
  and `getAutor()` uses an `EXISTS` subquery (line 53-54).
  The asymmetry is intentional (per the Javadoc) but
  means a typo in the user-typed `editorial` value still
  matches every book whose editor name contains the
  substring. For the "books with publisher 'Random'"
  query this returns Penguin Random House AND Random
  House Spain, which is rarely what the user wants.
  Fix: add a second `addConditionLikeWord` helper that
  wraps the value in `%` and prepends `+` for the
  MySQL full-text `+word*` syntax, or document the
  substring-match behaviour in the FilterDrawerPanel
  tooltip (the user is currently given no indication
  that the editor field is a substring match).

**[src/persistencia/LlibreDaoCore.java]** — [severity: LOW]
- `executeSQLFile` (line 303-356) uses
  `BufferedReader.read()` with a `char` cast. This
  silently breaks on any non-BMP character (a book
  title with a CJK ideograph or a 4-byte emoji in
  the notes field, both of which the SQL backup
  preserves as UTF-8). The character is read as two
  `char`s (the high/low surrogates) and the surrogate
  pair is rendered into the StringBuilder as-is,
  which is fine for round-tripping within Java, but
  the SQL string emitted to `executeUpdate` may not
  match what the DB originally stored if the
  driver does any normalization. Fix: read with
  `br.read()` and build the statement with
  `CodePoint` accumulation, or replace the
  character-at-a-time loop with
  `br.lines().forEach(...)` and re-implement the
  quote/comment tracking at the line level (which is
  actually what the upstream `Rfc4180Reader` does for
  CSV — the SQL parser could share the same state
  machine).

**[src/presentacio/ConfiguracioDialog.java]** — [severity: LOW]
- The 1-arg `ConfiguracioDialog(Frame parent)` (line 34)
  and 2-arg `ConfiguracioDialog(Frame parent,
  ConfiguracioDialogListener listener)` (line 36-38)
  constructors are dead. They both call the 3-arg ctor

## Calibration

1. **Weakest finding included.** The
   `BookImporter.importCalibre` pipe-delimiter parser
   finding (HIGH) is the strongest by failure-mode
   reachability — any Calibre library with a `|` in a
   book title silently corrupts 9 fields per affected
   book — but my evidence is the code, not a recorded
   test. To strengthen it: write a JUnit test with a
   `metadata.db` fixture containing one book with
   `title='C++|Java'`, run `importCalibre`, and assert
   the parsed `nom` does not contain the rest of the
   line. A real fixture would also pin the failing
   field index so a regression catches it
   deterministically.

2. **New class of issue to look for next run.** I
   didn't audit the **singleton + double-checked
   locking pattern in MainFrameControl.getInstance**
   carefully on prior runs (it was in the "single
   Swing init" boilerplate), and the racy
   check-then-construct is real. The same anti-pattern
