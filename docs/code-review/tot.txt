To strengthen: a thread-dump or jstack of `enRefrescarDades` against a synthetic 10k library, demonstrating the EDT block exceeds the 100 ms "Noticeable" threshold of the Swing repaint budget. As-is, the finding is correct but the magnitude is inferred, not measured.

2. **New class to look for next run.** I did not systematically audit `try-finally` blocks on `HttpURLConnection` and `InputStream` returns across the codebase beyond `ClientOpenLibrary` and `ServeiCoberta`. Several SwingWorkers call `cd.obtenirLlibreBlob(isbn)` via `LlibreBlobDao` and discard the byte[] return without `try-with-resources`-aware guarantees on the underlying stream. The next pass should trace every byte[]-returning JDBC path through `try { ... } finally { ... }` to confirm no stream/connection escape on the JDBC side (e.g., `LlibreDaoCore.obtenirRecentlyAdded` and `obtenirAll` both run unbuffered on the same `con` field without per-query resource isolation — fine for sync but a hazard under future async migrations).

3. **confidence: 70.** I read ~165 source files directly (no subagents), so every line cited in a finding is in the editor; CRITICAL and HIGH are grounded in concrete code paths (lock-ordering violation in `DelegatLlibre.actualitzarLlibre`, JDBC-on-EDT in `ControladorAccionsLlibre.enRefrescarDades`, static `synchronized rateLimit()` in `ClientOpenLibrary`, un-synced disk read in `ServeiCoberta.obtenirCachedBytes`).
