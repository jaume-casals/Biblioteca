# tot.txt — Biblioteca code review (per-file findings)

Severity guide: CRITICAL (bug / data loss / security / thread-safety / leak) · HIGH (anti-pattern, perf, PR-blocker) · MEDIUM (suboptimal but works) · LOW (most severe nitpick only if no CRITICAL/HIGH/MEDIUM). Cap: ≤3 findings per file, ranked.

## domini/

**[src/domini/AnalitzadorPrestatgeria.java]** — HIGH
- Line 39: `exportarToCsv(File, List<Llibre>, interficie.LectorPrestatgeria cd)` — `domini/` takes a parameter typed as `interficie.LectorPrestatgeria` and (lines 41, 43) calls `cd.obtenirAllLlistes()` / `cd.obtenirAllLlibreLlistaRows()`. The `domini` layer depending on the contract package (and transitively on `persistencia.LlibreLlistaRow` at line 43) is an MVC layering inversion; the contract should depend on `domini`, not the other way around. Fix: change the signature to `exportarToCsv(File, List<Llibre>, Map<Integer,Llista>, Map<Long,List<LlibreLlistaRow>>)` and have the caller (in `presentacio` or a thin `interficie` adapter) pre-compute the maps from the public domain API.
- Line 39 + line 64: method throws raw `Exception` (and the inner `catch (Exception e)` at 64 silently logs and skips rows). Throw `BibliotecaException` (or `IOException` for the writer path) so callers can react to "DB failure" vs "row bad data" differently; the `catch (Exception e) { LOG.log(...) }` should at least preserve the failing row's ISBN in the log message (currently it does: line 65).

**[src/domini/ControladorDomini.java]** — MEDIUM
- Line 47: `implements EscritorBiblioteca, EscritorLlibre, EscritorPrestatgeria, EscritorEtiqueta, EscritorPrestec` — the façade sits in `domini` and satisfies five interfaces from `interficie/`. The pattern works, but the cleanest arrow is `interficie → domini` (adapter in `interficie` that delegates to the façade). Either way, the five `Escritor*` interfaces duplicate method signatures (see `interficie/EscritorBiblioteca.java:26-50` finding) and `ControladorDomini` ends up implementing each method twice via the merged interface — extracting the duplication would let `ControladorDomini` implement the new `EscritorBiblioteca` once.

**[src/domini/EspecificacioOrdenacio.java]** — LOW
- Lines 81-85: `comparators()` allocates a fresh `HashMap` on every call. `DelegatLlibre.SORT_BY` consumes this once at class init, but a one-time cache (`private static final Map<String, Comparator<Llibre>> COMPARATORS = comparators();`) would document the intent, eliminate the per-call allocation, and let the method be `static` private.

**[src/domini/Llibre.java]** — MEDIUM
- Lines 149-166: `obtenirAutor()` joins on every call (line 150 `String.join(", ", autors)`), `obtenirAutors()` (line 154) returns a defensive `new ArrayList<>(autors)` copy every call, and `afegirAutorNom` (line 160) re-joins the whole list on every insertion (O(N²) when bulk-loading N authors from the DB). For a 10k-book table that re-renders on every filter change, `obtenirAutor()` alone is O(N×M). Fix: keep `autor` cached and only recompute on `posarAutors` / `clearAutors`; return `Collections.unmodifiableList(autors)` from `obtenirAutors()` since the docstring at the top of the class says `Llibre` is mutated in place anyway. Also line 31 uses fully-qualified `new java.util.ArrayList<>()` — add an import and shorten.

**[src/domini/Llista.java]** — CRITICAL
- Lines 60-74: `equals` (line 63: `return this.id == ((Llista) o).id;`) returns `true` for any two transient `Llista` instances (id=0) regardless of `nom`, but `hashCode` (line 73) mixes in `nom` for id=0. Consequence: `new Llista(0, "A")` and `new Llista(0, "B")` are `equals()` yet hash to different buckets, so a `HashSet<Llista>` contains both as separate entries and a `HashMap<Llista, …>` lookup will silently miss. The author's comment at lines 68-72 acknowledges the transient-vs-persisted split but applies the fix only in `hashCode`, not in `equals` — the contract violation goes the other way (equal objects, unequal hashes). Fix in `equals`: `if (id == 0 || other.id == 0) return Objects.equals(nom, other.nom); return id == other.id;`.

**[src/domini/facade/DelegatLlibre.java]** — MEDIUM
- Lines 130, 155, 160, 202 (and the `cercarKey` helper at 262-265): every `binarySearch` constructs a throwaway `Llibre` (with blank `nom`/`autor`, default `any`/`valoracio`/`preu`, etc.) just to leverage `ISBN_COMPARATOR`. This is an allocation per search and a correctness landmine — if `ISBN_COMPARATOR` is ever updated to inspect another field, a transient key with default values will mis-order. Fix: `static int compareByIsbn(Llibre l, long isbn) { Long i = l.obtenirISBN(); return i == null ? -1 : Long.compare(i, isbn); }` and `Collections.binarySearch(state.bib(), isbn, (l, k) -> compareByIsbn(l, (long) k))` — no allocation, no hidden coupling.

**[src/domini/facade/DelegatPrestatgeria.java]** — CRITICAL
- Line 43: `obtenirAllLlistes()` returns `new ArrayList<>(state.llistes())` without taking the lock — but every other method that touches `state.llistes()` or `state.llistesById()` (`afegirLlista`, `eliminarLlista`, `reanomenarLlista`, `moureLlistaUp/Down`, `posarLlistaColor`, `swapLlistesOrdreLocked`) does so under `withLock` / `withLockReturning`. `new ArrayList<>(state.llistes())` triggers an `ArrayList.Itr` over the live list; a concurrent `afegirLlista` / `eliminarLlista` / `moureLlistaUp` / `moureLlistaDown` will throw `ConcurrentModificationException` or return a torn snapshot. The class-level Javadoc (lines 19-34) explicitly claims "el lock d'estat es manté durant tota la seqüència pre-comprovació → persistència → mutació en memòria" — this method is the lone exception. Fix: `return state.withLockReturning(() -> new ArrayList<>(state.llistes()));`.

**[src/domini/facade/StateContext.java]** — MEDIUM
- Lines 91-98 (`netejarAll`) hand-clears the three lists and then calls `rebuildIdIndexesLocked()` (lines 101-106), which iterates the now-empty lists to rebuild the maps. The author's own comment (lines 86-90) admits the duplication: "avui els dos camins buiden els mapes directament, però la crida centralitzada fa que futures addicions — p.ex. un nou mapa d'índex per id — siguin un canvi d'una sola línia en lloc d'una cerca de dues línies." Either pass empty lists to `replaceAll(empty, empty, empty)` so both paths share one body, or have `rebuildIdIndexesLocked` clear the lists itself. Don't ship a comment promising single-source-of-truth and leave the duplication in.

## persistencia/

**[src/persistencia/ConnexioServidor.java]** — MEDIUM
- Lines 391-392: `props.setProperty("password", new String(password))` materializes a `String` from the `char[]` — exactly the conversion the Javadoc at lines 126-135 (and the rest of the project) is trying to avoid. At minimum, zero the local copy: `String pwd = new String(password); java.util.Arrays.fill(password, '\0'); props.setProperty("password", pwd);` so the caller's intent ("don't leave a `String` on the heap after I return") is at least partially honored.
- Lines 167-202 (`crearDatabase`): if `runMigrations()` (or any later step) throws after `connectViaDriver` succeeded, `con` is left open and the `catch` block only wraps the exception — no `tancarConnection()` in a `finally`. The singleton is then abandoned with a live JDBC `Connection` that pins the H2 file lock. Fix: do the connect in a helper and assign `con` only after `runMigrations()` succeeds, or wrap the body in try/finally that calls `tancarConnection()` on the way out.

**[src/persistencia/JarLocator.java]** — LOW
- Lines 131-136 (`recordAndReturn`): `teJars.test(lib)` is invoked twice per candidate directory (once for the diag label, once for the return decision). The predicate does `list((d, n) -> n.endsWith(".jar"))` on every call — wasted IO. Store the boolean: `boolean has = teJars.test(lib); String label = has ? " HAS_JARS" : lib.isDirectory() ? " no-jars" : " missing"; …; return has ? lib : null;`.

**[src/persistencia/LlibreDaoCore.java]** — HIGH
- Lines 287-300 (`obtenirDbSizeBytes`): `.replaceAll("\\s+", "")` is applied to the H2 path derived from `con.getMetaData().getURL()`. A profile name like `"my library"` (allowed by `sanitizeH2Profile`, which only rejects `/`, `\`, `..`) produces URL `jdbc:h2:/home/j/.biblioteca/my library;...`, which the regex then folds to `/home/j/.biblioteca/mylibrary`. The lookup `new File(path + ".mv.db")` silently misses the real DB and returns `-1`. The caller cannot distinguish "no DB" from "size 0". Drop every `.replaceAll("\\s+", "")` (the `trim()` already handles leading/trailing whitespace) and parse with `URI` + `substring` up to the first `;`.

**[src/persistencia/LlibreSearchDao.java]** — MEDIUM
- Line 27 + 74 (`search`): only `pageSize` is validated (`>= 0`); `offset` is appended raw into `LIMIT ? OFFSET ?`. A negative `offset` produces `... LIMIT 50 OFFSET -5`, which both H2 and MariaDB reject with a generic syntax error that the caller then wraps in `BibliotecaException`. Add `if (offset < 0) throw new IllegalArgumentException("offset must be >= 0; got " + offset);` next to the `pageSize` check, or pass `offset` as a `setInt` parameter (the `LIMIT`/`OFFSET` numeric values can be parameterized in both engines).

**[src/persistencia/LlistaDao.java]** — MEDIUM
- Lines 105-125 (`obtenirLlibres`): a 27-column projection is hardcoded in the SQL string instead of reusing `LlibreDaoCore.Columna` / `buildSelect`. The whole point of `LlibreDaoCore.LLIBRE_COLUMNS_L` and the `buildSelect(...)` helper is to make adding a column a one-line change; this query silently bypasses that and will drift the next time someone adds a field. Fix: extend `LlibreDaoCore.buildSelect` to take an `Override` map (`valoracio→ll.valoracio`, `llegit→ll.llegit`) so the join case consumes the same builder, or expose a small `LlibreDaoCore.selectLlibreWithShelfOverride()` helper.

**[src/herramienta/ExportadorLlibres.java]** — HIGH

**[src/herramienta/I18n.java]** — MEDIUM
- Lines 628-629: the replacement loop mutates `raw` in place: `raw = raw.replace("{" + i + "}", String.valueOf(args[i]));`. `String.replace(CharSequence, CharSequence)` is non-replacing per replacement of the *substring* — so an `args[i]` whose stringified value contains a later placeholder (e.g., the user passes `"page {1}"` as `args[0]`, then `args[1]` gets substituted into it on the next iteration) is silently corrupted. Build the output by composing the segments around placeholders, or escape `{`/`}` in the arg values first. Low blast radius (the format is i18n text, not arbitrary user data) but real.

**[src/herramienta/ImportadorLlibres.java]** — MEDIUM
- Lines 85-90: the `stderrDrain` thread is started after `proc.getErrorStream().transferTo(...)` is called inside the lambda, but the lambda only runs when the drain thread is alive. There's a small window between `Runtime.exec` returning and the daemon thread actually starting `transferTo` where the OS pipe buffer (typically 64 KB on Linux) can fill and block the child. Start the drain thread before the first `readLine()` loop, or move the drain start to right after line 84.
- Line 135: `proc.waitFor(30, TimeUnit.SECONDS)` returns `true` on normal exit, but the exit value is never read. If `sqlite3` exits non-zero, we still parse its stdout and report success. Add `if (proc.exitValue() != 0) throw new Exception("sqlite3 exit " + proc.exitValue());` after the read loop completes.

## Calibration

1. **Weakest finding.** The I18n placeholder-replacement bug at `herramienta/I18n.java:628-629` is the weakest: I argued `String.replace(CharSequence, CharSequence)` is non-replacing per replacement of the *substring*, but `String.replace` actually does a literal-replace of the *first* occurrence and returns a new `String` — the issue is more subtle (the inner `replace` re-scans the *new* `raw` on the next iteration and may inject placeholders that weren't in the input). The behavior is also bounded by the fact that `args` values today come from numeric/format outputs, not user input. Stronger evidence would be a unit test demonstrating the corruption with a realistic arg, plus a count of call sites in `presentacio/` that pass user-derived strings to `I18n.t`.
